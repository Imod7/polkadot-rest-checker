How to read this report

Each benchmark was run multiple times. The median is used for comparison because it ignores outliers (e.g., a single slow run caused by GC, OS scheduling, or network jitter).

If the average is significantly lower than the median, it means some runs were much slower than typical — this indicates high variance. High variance is itself a performance signal: it may indicate garbage collection pauses (common in Node.js/TypeScript runtimes like Sidecar), memory pressure, or unstable response times under load. A tool with lower variance delivers more predictable, reliable performance.

stdev (standard deviation) quantifies the spread. Lower stdev = more consistent runs.

Glossary

Stdev (Standard Deviation) in Request Duration

Stdev measures how spread out the request latencies are. A low stdev means most requests complete in a similar time — the service is predictable. A high stdev means some requests are much slower than others — the service is inconsistent.

A service can have a lower average latency but higher stdev. This means it's usually faster, but occasionally has slow outliers. Common causes: metadata validation on first requests (subxt re-validates storage paths against runtime metadata), connection pool warmup (new WebSocket connections are slower than reused ones), node-side cache misses (first access to a storage key hits disk), or thread scheduling (multi-threaded runtimes like Tokio have more jitter than single-threaded event loops like Node.js).

The stdev is computed across all requests in the run (often tens of thousands). Look at P50 vs P99 to see the actual spread — if P99 is much higher than P50, the slow outliers are what's driving the stdev up.

Coefficient of Variation (CV)

CV = stdev / avg, expressed as a percentage. It measures how large the spread is relative to the average. This is more meaningful than raw stdev when comparing two services with different average latencies.

Example: Service A has avg 50ms, stdev 200ms (CV = 400%). Service B has avg 500ms, stdev 265ms (CV = 53%). Raw stdev says A is "more consistent" (200 < 265), but A's responses swing ±4x its average — that's wildly unpredictable. B's variation is only ±53% of its average — much more proportionally stable.

Lower CV = more consistent. A CV under ~50% is typical for a stable service under load. A CV over 100% means the stdev exceeds the average — response times are highly variable.

Why is Max Latency So High?

The max is the single slowest request out of the entire run — one request out of potentially hundreds of thousands. It is almost always an outlier caused by:

• Cold start effects — the very first request may trigger metadata fetching, connection establishment, or cache population
• Node-side storage misses — a particular block or account that wasn't in the RPC node's memory cache, forcing a disk read
• GC pauses — garbage collection in Node.js (sidecar) or memory allocation spikes in Rust (rest-api)
• Network jitter — a brief WebSocket hiccup between the API and the node
• OS scheduling — the process was briefly preempted by the kernel

The max is not representative of user experience. Use P99 (worst 1% of requests) or P99.9 (worst 0.1%) for a realistic worst-case. To investigate what caused a specific slow request, you would need request-level logging with timing breakdowns (time in RPC call vs serialization vs network).

CPU per 1K Requests — Why Throughput Matters

Comparing raw CPU usage between two services is misleading if they handle different amounts of traffic. A service processing 2x more requests should use more CPU — that's expected.

CPU% per 1K RPS normalizes by throughput: it divides average CPU usage by requests per second. Lower = more efficient. However, this metric has a subtle bias: at higher throughput, CPU per request can increase due to contention (lock contention, cache pressure, context switching). The lower-throughput service may appear "more efficient" simply because it never reached the load level where these effects kick in.

To get a fair comparison, both services should ideally be tested at the same request rate (using wrk2's constant-rate mode) rather than at their respective maximums.

Memory per 1K Requests — Why Throughput Matters

Similar to CPU, comparing raw memory usage is misleading at different throughput levels. A service handling more concurrent requests needs more memory for connection buffers, in-flight request state, and response serialization buffers.

MB per 1K RPS normalizes by throughput. A service that uses 200MB to serve 1000 RPS is more memory-efficient than one that uses 150MB to serve 500 RPS (0.2 MB/1K vs 0.3 MB/1K).

Note: some memory usage is fixed overhead (runtime, metadata cache, connection pools) that doesn't scale with request rate. This fixed cost is amortized better at higher throughput, which can make higher-throughput services appear more memory-efficient even if their per-request allocation is similar.
